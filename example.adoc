== Введение

Добро пожаловать в документацию по *CloudSync* — современному решению для синхронизации данных между локальными серверами и облачным хранилищем. Данное руководство описывает процесс установки, настройки и устранения неполадок.

=== Основные возможности

* 🚀 Мгновенная синхронизация файлов любого размера.
* 🔒 Сквозное шифрование данных (AES-256).
* 🔄 Поддержка инкрементальных резервных копий.
* 🌐 Работа через прокси-серверы и брандмауэры.

[NOTE]
Эта версия программного обеспечения требует как минимум **Java 17** или **Python 3.9** для работы вспомогательных скриптов.

== Установка

Процесс установки зависит от вашей операционной системы. Выберите соответствующий раздел ниже.

=== Требования к системе

Перед началом убедитесь, что ваша система соответствует следующим критериям:

[cols="1,2", options="header"]
|===
| Компонент | Минимальные требования

| ОЗУ
| 4 ГБ (рекомендуется 8 ГБ)

| Дисковое пространство
| 500 МБ для приложения + место под данные

| Сеть
| Стабильное соединение от 10 Мбит/с
|===

=== Установка на Linux

Для дистрибутивов на базе Debian/Ubuntu выполните следующие команды в терминале:

[source,bash]
----
sudo apt update
sudo apt install cloudsync-core
cloudsync-cli init --config /etc/cloudsync/config.yaml
----

Для пользователей Red Hat/CentOS используйте менеджер пакетов `yum` или `dnf`:

[source,bash]
----
sudo dnf install cloudsync-core
systemctl enable cloudsync
systemctl start cloudsync
----

[TIP]
Используйте флаг `--verbose` при инициализации, чтобы получить подробный лог процесса настройки.

=== Установка на Windows

1. Скачайте установщик `.msi` с официального сайта.
2. Запустите файл от имени администратора.
3. Следуйте инструкциям мастера установки.
4. Перезагрузите систему после завершения.

== Конфигурация

Основной файл конфигурации находится по пути `/etc/cloudsync/config.yaml` (Linux) или `C:\ProgramData\CloudSync\config.yaml` (Windows).

=== Пример конфигурации

Ниже приведен пример корректного файла настроек:

[source,yaml]
----
server:
  host: "sync.cloudsync.example.com"
  port: 443
  protocol: "https"

auth:
  method: "oauth2"
  client_id: "cs_client_12345"
  token_refresh_interval: 3600

sync:
  interval: 300 # секунды
  exclude_patterns:
    - "*.tmp"
    - ".git/"
    - "node_modules/"
  
  encryption:
    enabled: true
    algorithm: "AES-256-GCM"
----

=== Параметры окружения

Некоторые настройки можно переопределить через переменные окружения:

|===
| Переменная | Описание | Значение по умолчанию

| `CS_LOG_LEVEL`
| Уровень логирования (DEBUG, INFO, WARN, ERROR)
| `INFO`

| `CS_MAX_THREADS`
| Количество потоков для обработки файлов
| `4`

| `CS_PROXY_URL`
| Адрес прокси-сервера
| _нет_
|===

== Использование

После настройки сервис готов к работе. Вы можете управлять им через CLI или веб-интерфейс.

### Базовые команды CLI

* `cloudsync status` — проверить текущее состояние службы.
* `cloudsync sync --force` — принудительно запустить полную синхронизацию.
* `cloudsync logs --follow` — просмотр логов в реальном времени.

Пример вывода команды статуса:

[source,text]
----
$ cloudsync status
Status: RUNNING
Uptime: 4d 12h 30m
Last Sync: 2 minutes ago
Files Pending: 0
Errors: 0
----

== Устранение неполадок

Если вы столкнулись с проблемами, проверьте следующие распространенные сценарии.

=== Ошибка подключения к серверу

Если вы видите ошибку `Connection Refused`, убедитесь:

1. Брандмауэр не блокирует порт 443.
2. Сертификаты SSL действительны.
3. Прокси-сервер настроен корректно.

[WARNING]
Не отключайте проверку сертификатов в производственной среде! Это создает уязвимость безопасности.

=== Проблемы с правами доступа

Убедитесь, что пользователь, от имени которого запущен сервис, имеет права на чтение и запись в целевые директории.

[source,bash]
----
chown -R cloudsync:cloudsync /var/data/sync
chmod 750 /var/data/sync
----

== Часто задаваемые вопросы (FAQ)

[float]
=== Можно ли синхронизировать скрытые файлы?
Да, но по умолчанию они исключены. Добавьте правило `-!.hidden*` в секцию `exclude_patterns`, чтобы включить их.

[float]
=== Как изменить интервал синхронизации?
Отредактируйте параметр `interval` в файле `config.yaml` и перезапустите службу.
